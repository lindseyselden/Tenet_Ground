## What changed
Rule #01 (Technology Presence) was rewritten to govern how technology *behaves* rather than whether
screens exist. Screens and interface surfaces are now explicitly permitted when recessive — embedded
flush into the architecture, data flat and editorial, architecture still dominant. The "avoid" list
shifted from "screens, displays" to the performing forms: floating/free-standing displays, device
chrome, glowing or atmospheric screens, command center, futuristic UI. The pass/fail test is
unchanged.

## Why
The literal "no screens" reading already contradicted the canon: the Ground Hall elevator type system
and entryway signage are screens, established in the typographic extension. The rule's real intent
was always recessiveness — its pass/fail test is "architecture before technology," which a flush,
editorial screen passes. The rule had over-indexed on the token "screens" because the failure mode it
fights (command-center, generic SaaS) wears screens. A UI/status-screen WIP forced the question, and
the answer refines the rule rather than weakening it — the same productive-drift pattern as
orthogonal-over-biomorphic.

## What this supersedes
Amends Rule #01's literal screen-ban; the pass/fail test and recessiveness rationale stand.
Asset-level specs are unaffected — the Ground Ring keeps "no screen" and the Ground Hall keeps "no
command-center monitors," because those are correct for those objects. Unblocks the UI workstream,
which "no screens" had made paradoxical.
